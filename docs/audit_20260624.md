## Section 1 — Architecture & Data Flow

The live architecture is clean and consistent:
`raw CSV → precompute_serie_a.py → {parquet in data/ready/} → data_loader._read_parquet() (lru_cache) → callback → component`.
`app.py` warms the match cache for all seasons at startup and rebuilds cross-season aggregates when per-season parquets are newer (`_ensure_fresh_data()`).

### 1.1 — Data flow maps (three representative chains)

```
PPDA season summary:
[serie_a_{season}/events/*.csv] → build_ppda_table() [ppda.py]
  → precompute_season() step 6 → {ppda_{season}.parquet}
  → data_loader.load_ppda_summary() → _read_parquet (lru_cache)
  → team_detail_callbacks.update_chart/ppda figures → PPDA bar/scatter UI

Chance Creation season aggregate (Opponent Analysis):
[*.csv] → analyse_chance_creation() [chance_creation.py]
  → precompute_season_offensive() → {shots_{season}.parquet, offensive_summary_{season}.parquet}
  → season_offensive_summary.compute_season_chance_creation() (cache_get/set, reads parquet)
  → analysis_callbacks._cc_* lazy loaders/modals → opponent_offensive_phase cards

Player Analysis per-match stats (Match Analysis):
[one match CSV] → analyse_player_analysis(Path) [player_analysis.py]
  → cache_get/set keyed by (stem, mtime, pv_token) — bundle cached in-memory
  → player_analysis_callbacks → player_analysis_cards UI
```
All three chains are complete. No live UI component reads raw CSV directly at request time (see 5.1).

### 1.2 — Precompute completeness (generator ↔ consumer)

Every parquet stem read by `data_loader.py` is generated by `precompute_serie_a.py` **except two**:

- **`goal_distribution_{season}.parquet`** — read by `load_goal_distribution()` (data_loader.py:530) but **never written by any precompute function**. No files exist on disk. Behaviour: the loader always falls through to `compute_goal_distribution()` (live raw-CSV compute). **MEDIUM** — graceful (no crash), but defeats the precompute design for this one chart; every request recomputes from raw.
- **`formation_positions_{season}.parquet`** — read by `load_formation_positions()` (data_loader.py:485). **No `_save_parquet` / `to_parquet` writer exists anywhere in the repo**, yet 5 files (~45 KB each) sit in `data/ready/`. They were produced by a now-removed step or a notebook. Behaviour: loader returns an empty DataFrame on miss (no live fallback). **MEDIUM** — if these files are ever deleted, the formation-position overlay silently goes blank and cannot regenerate.

No **zombie** parquets (generated-but-unread): every other written stem has a consumer.

### 1.3 — Cache coherence

- `invalidate_cache()` clears both `_read_parquet.cache_clear()` and `load_season_matches_cached.cache_clear()`, and is called after every `precompute_season()` in `ensure_ready_data()` and after `build_league_summary()` in `app.py`. Correct.
- The TTL cache (`utils/caching.py`, 1 h) is **not** flushed by `invalidate_cache()` — only `_read_parquet`/match caches are. The TTL cache holds `season_offensive_summary` benchmark dicts and modal payloads keyed by season. **MEDIUM (latent):** if a season is re-precomputed mid-session without restart, TTL-cached benchmark/origin dicts can serve stale data for up to 1 h. In practice precompute runs at startup, so the window is small; flagged for completeness.

### 1.4 — Protected function integrity

- **`canonical_name()`** (team_mapping.py:122) — unchanged since initial commit; trivial dict lookup. No caller passes unexpected formats. Intact.
- **`classify_attack_origin()`** (chance_creation.py:210) — intact and internally consistent. **Discrepancy to confirm (not a regression):** the actual priority order in code is **Through Ball → Set Piece → Individual Play → High Regain → Cut Back → Cross → Combination**. The audit brief stated the expected order as *Set Piece → High Regain → Counter → Cross → Through Ball → Combination → Individual Play*. The code order is heavily documented as intentional (Through Ball qualifier is authoritative and must not be overridden by set-piece proximity; Cut Back precedes Cross by design). There is no "Counter" origin in the code. **LOW** — treat the brief's ordering as outdated relative to source; confirm with Riccardo that the current order is the intended one.

### 2.3 — Circular dependency check

This is a **deliberate lazy-load pattern** (skeleton first, heavy sections async), not an accidental serial chain. **LOW/acceptable.** The only real serial depth is 2 (view → trigger → sections); sections then run concurrently.

## Section 5 — Performance Risks

### 5.1 — Raw CSV reads inside callbacks — **NONE (clean)**
`grep` for `read_csv`/`open(` across `src/callbacks/` returns **zero matches**. The precompute migration is fully intact at the callback layer. The Match Analysis / Player Analysis callbacks call `analyse_player_analysis(Path(csv))`, which reads **one** match CSV but is wrapped by an explicit `cache_get/cache_set` bundle cache keyed by `(stem, mtime, pv_token)` (player_analysis.py:819-854). Single-match, cached — acceptable.

### 5.2 — Unguarded expensive computations
- Season-aggregate modals (`compute_season_gk_buildup`, `compute_season_chance_creation`, `compute_league_offensive_benchmarks`, `compute_league_clean_sheets`) read **parquet** and are guarded by `cache_get/cache_set` (some inline in the callback, e.g. `_cc_xg_modal_graph`). **OK.**
- A few pressing/castle/CC league-comparison modals call `load_league_*_summary(season_key)` on every open without an explicit `cache_get` wrapper at the callsite — but those loaders ultimately hit `_read_parquet` (lru_cache) or `cache`-decorated readers, so the cost is a cached parquet filter. **LOW.**

### 5.3 — Large data through `dcc.Store`
The Opponent Analysis sections push per-section payloads into stores (`opp-season-ft-entries-store`, `-press-store`, `-castle-store`, `-cc-conceded-store`, `-sp-store`, `-trans-*-store`, `-player-store`). Some carry lists (FT `entries`, corner `corners`, shot arrays) serialised to the browser. For a full season these are bounded (hundreds–low-thousands of rows) but non-trivial. **MEDIUM** — monitor the FT `entries` and corner `corners` stores on heavy teams; they are the largest client-side payloads. Not a correctness risk.

### 5.4 — Missing caching wrappers
`analyse_player_analysis` and `_load_match_cached` are correctly cached. `_read_parquet` is `lru_cache(maxsize=64)` — with 110 ready parquets across 5 seasons, **64 may be too small** if a user browses many seasons/sections in one session, causing evictions and re-reads. **LOW** — consider raising `maxsize` to ~128.

## Section 6 — Debt & Fragility Hotspots

### 6.1 — Hardcoded magic values — **HIGH**
Pitch-coordinate thresholds (`83.33` penalty-box x, `66.67`/`16.67` third boundaries, etc.) are hardcoded across **28 files / ~106 occurrences** in `src/analytics/` and `src/components/`. There is no central `PITCH` constants block in `config.py`. Any change to zone definitions requires a 28-file edit and is error-prone. Recommend a `config.py` (or `styling/pitch_utils.py`) constants group: `PEN_BOX_X = 83.33`, `FINAL_THIRD_X = 66.67`, `DEF_THIRD_X = 33.33`, etc. Season strings appear as literals ~23 times (mostly `"2024_2025"` fallback defaults in the dead `tabs_callbacks`); the live default is centralised as `DEFAULT_SEASON` in `config.py`. **HIGH** for coordinates, **LOW** for season strings.

### 6.2 — Inconsistent PPDA definitions — **OK (by design, verified)**
Two definitions coexist intentionally:
- Team Overview / `ppda.py` `build_ppda_table` → ball-recoveries denominator (used by `load_ppda_summary` → team_detail).
- Match/Opponent Analysis → `defensive_pressing.analyse_defensive_pressing` accumulates `ppda_num/den_*` (tackles + interceptions + fouls + challenges) into `pressing_summary`.
Each callsite reads the matching source for its context (team_detail uses `ppda_{season}`; opponent pressing uses `pressing_summary_{season}`). No cross-wiring found. Documented divergence is respected.

### 6.3 — Error-handling gaps
- `data_loader._read_parquet` and `_is_readable_parquet` wrap reads in try/except and degrade gracefully. Good.
- `tabs_callbacks.render_tab_content` and `render_match_report_sub` have try/except (but are dead).
- The **live** lazy section loaders in `analysis_callbacks.py` (`_opp_load_*`) call `build_*_section(...)` with **no try/except**. If a builder raises (e.g. malformed parquet, unexpected None), the `dcc.Loading` spinner spins forever / Dash surfaces a callback error rather than a friendly message. **MEDIUM** — wrap each `build_*_section` body in try/except returning a `dbc.Alert`, mirroring the dead `tabs_callbacks` pattern.
- `team_detail_callbacks` figure builders read parquet via loaders that return empty frames on miss; figures then render empty rather than crash. Acceptable.

### 6.4 — Theme consistency gaps
- Live hardcoded background hexes: `src/components/tables.py` (`#2c3e50`, `#1e1e1e`, `#252525` — but the file is **dead**) and `src/registry/loaders.py` DataTable styling (`#2c3e50`/`#1e1e1e` — also dead). The only live hardcoded `paper_bgcolor` hex is in `multi_season_standings_v1.py` (also dead). **Net: no live theme-breaking hardcodes** once the dead cluster is removed. **LOW.**
- The clientside theme patcher (`theme_callbacks.py`) correctly skips `.pitch-dark-container`/`.formation-pitch` (dark-by-design) and patches everything else to transparent + CSS-variable-driven font colors. Server-drawn zone fills (transitions lights, corner maps, pressing heatmaps) are rebuilt on theme change via dedicated callbacks. Sound design.

### 6.5 — Documented known issues — status
- **`_is_set_piece_event()` team-filter bug — FIXED.** The function now takes `attacking_team` (chance_creation.py:311) and only counts a restart as a set piece when it belongs to the attacking team (guard at lines 343+); `_check_set_piece` passes `attacking_team` at lines 424/430/488. The GW13/GW17/GW19 misclassification (opponent restart inside the 15 s window) is resolved. **CLOSED.**
- **`GRAPH_REPORT.md` outdated** — confirmed not authoritative; this audit relied on source files throughout. It correctly does not reflect the dead `tabs/registry` cluster. Treat as navigation aid only.
